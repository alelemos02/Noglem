## Problem Statement

Engenheiros de instrumentação/automação em projetos industriais (O&G, petroquímica, construção pesada) precisam emitir pareceres técnicos de fornecedor (PATEC) para cada equipamento do projeto. Um projeto simples demanda 700-1.000 horas de engenharia — feito manualmente em Excel, Word e e-mail.

JulIA reduz esse processo para ~30 horas (3-4 minutos automatizados + revisão humana com o fornecedor). O executor atual do trabalho não tem ferramenta especializada: usa planilhas customizadas sem padronização, sem automação, sem audit trail.

Ferramentas genéricas de procurement (SAP Ariba, Enverus RFx) gerenciam o **processo** mas não entendem o **conteúdo técnico** (ATEX, SIL, IP, condições de processo, especificações de instrumento). O engenheiro ainda avalia centenas de parâmetros manualmente. JulIA é o primeiro sistema que avalia o conteúdo técnico em si.

## Demand Evidence

- **Founder é o usuário:** 18 anos em instrumentação/automação industrial. Constrói e usa as ferramentas.
- **Empregador quer usar internamente:** A empresa onde o founder trabalha está estudando adotar as ferramentas, mas tem objeções de segurança/privacidade de dados com IA.
- **Techint (EPC):** Fúlvio, coordenador de projetos, quer o PATEC. Atraso no parecer = atraso na obra = descredibilidade profissional para ele. É o usuário com mais dor e maior urgência identificada.
- **Outros engenheiros:** Mostrados a ferramenta nas últimas semanas — "extremamente interessados."
- **Bloqueio principal:** Medo de IA vazar dados proprietários de projeto para terceiros. "IA ainda é tabu" no setor industrial brasileiro.

## Status Quo

Processo atual para emissão de um PATEC simples:
1. Receber datasheet do fornecedor em PDF
2. Abrir planilha Excel customizada (sem padrão entre empresas)
3. Transferir manualmente parâmetros: tag, faixa, exatidão, classificação ATEX, SIL, IP, material de processo, conexão, certificações
4. Comparar com especificação técnica do projeto (outro PDF/Excel)
5. Escrever parecer em Word com justificativas técnicas
6. Enviar por e-mail para revisão
7. Repetir para cada instrumento do projeto

**Custo:** ~700-1.000 horas por projeto simples. Ao custo médio de R$180/hora = R$126.000-R$180.000 em labor **só nessa atividade**.

O concorrente real é o Excel — não há software especializado no mercado brasileiro.

## Target User & Narrowest Wedge

**Usuário primário:** Coordenador de engenharia de instrumentação em construtoras/EPCs (empresas como Techint, CB&I, Worley, Petrobras Engineering).
- Título: Coordenador de Instrumentação / Gerente de Engenharia
- Dor: Cronograma e credibilidade. Atraso no PATEC = atraso na obra.
- Usuário nomeado: Fúlvio (Techint)

**Wedge mínimo pago:**
- Um gerador de PATEC para **transmissores de pressão** (maior volume, mais padronizados)
- Input: datasheet do fornecedor em PDF + especificação técnica do projeto
- Output: parecer técnico no template exato da empresa cliente (Word/PDF)
- Completude de campo: ~15 parâmetros técnicos com veredicto PASS/FAIL/REVIEW

O template da empresa cliente é o moat real — quem tem o template Techint não precisa mais do Excel.

## Constraints

- Alexandre ainda está empregado — não pode usar dados proprietários do empregador sem formalização
- A empresa do founder quer usar, mas não assinou nada ainda; risco de revendicar propriedade intelectual se usada internamente sem contrato
- IA é tabu no setor: objeção de segurança de dados é universal e consistente

## Premises (confirmadas pelo founder)

1. **PATEC é o produto principal** — não o extrator de P&ID nem outros tools
2. **Cliente primário é EPC/construtora** (Techint, etc.), não o engenheiro individual
3. **Caminho mais rápido para receita é Techint/Fúlvio** — a empresa do founder é referência, não receita imediata (a menos que formalizada como contrato arm's-length >R$50k/ano em 60 dias)
4. **Concorrência real é o Excel** — ferramentas genéricas de procurement não são ameaça
5. **Distribuição inicial é B2B via relacionamento direto** — não SaaS de self-service

*Nota: Premissa 3 foi revisada após cross-model challenge. O agente independente identificou corretamente que venda para empregador ancora em zero se não for formalizada.*

## Cross-Model Perspective

**Segunda Opinião (Claude subagent):**

Linha mais reveladora: "Fúlvio at Techint — missing a PATEC deadline means project delay and professional credibility loss." Tudo mais é soft. Se Fúlvio pagar ou defender dentro da Techint, existe empresa.

Premissa desafiada: "Venda interna para empregador é o caminho mais rápido." O subagente corretamente identificou que vendas internas ancoram em zero. Techint é o primeiro cliente real.

Protótipo em 48h proposto: PATEC generator para transmissores de pressão — upload de dois PDFs, extração estruturada com Pydantic, tabela PASS/FAIL/REVIEW, output no template Techint.

## Approaches Considered

### Approach A: "Piloto Fúlvio"
Build minimal PATEC para um tipo de equipamento, piloto com Techint, endereça privacidade no discurso.
- Esforço: S | Risco: Médio
- Pros: mais rápido, Fúlvio identificado
- Cons: objeção de dados pode bloquear antes do contrato

### Approach B: "On-Premise First" (escolhida)
Deploy dentro da infraestrutura do cliente. Dados nunca saem. Posicionamento: "IA industrial que nunca vê seus dados."
- Esforço: M | Risco: Médio
- Pros: remove objeção permanentemente, premium pricing, moat de instalação
- Cons: setup mais complexo por cliente

### Approach C: "Coordenador de Inovação"
Negociar cargo interno no empregador, construir com dados reais.
- Esforço: XL | Risco: Alto
- Pros: acesso a dados reais
- Cons: lento, risco de lock-in intelectual

## Recommended Approach

**On-Premise First (Abordagem B)**, em duas fases:

**Fase 0 — Validação (target: 2026-05-16, sem código novo):**
Ligar para Fúlvio. Mostrar o PATEC atual. Pedir: (a) o template exato da Techint para pareceres técnicos, (b) um dataset anonimizado de um projeto real. Essa conversa em 45 minutos vale mais que 3 meses de código.

**Fase 1 — Protótipo (2-3 semanas, target: 2026-06-06):**
- PATEC generator para transmissores de pressão
- Upload: datasheet do fornecedor + spec do projeto
- Extração estruturada (Claude API + Pydantic), ~15 campos
- Tabela PASS/FAIL/REVIEW com regras **determinísticas** para o veredicto. Exemplos de critérios mínimos:
  | Parâmetro           | Regra de PASS                              |
  |---------------------|---------------------------------------------|
  | Faixa de medição    | Faixa fornecedor ≥ faixa especificada       |
  | Classificação ATEX  | Zona fornecedor = zona especificada         |
  | Nível SIL           | SIL fornecedor ≥ SIL especificado          |
  | Grau IP             | IP fornecedor ≥ IP especificado             |
  | Material em contato | Material fornecedor ∈ lista de materiais aceitos |
  | Conexão de processo | Conexão fornecedor = conexão especificada   |
- Output: parecer no template da Techint (Word/PDF)
  - **Se o template não for obtido com Fúlvio até 2026-05-16**: usar template genérico ABNT como placeholder e marcar como "Draft — template Techint pendente"
- Stack: FastAPI + Next.js existentes. Sem Clerk, sem multi-tenancy, sem billing ainda.
- **Nota sobre Claude API:** Fase 1 usa Claude API (Anthropic) para extração. Isso significa que os documentos saem da rede durante esse estágio. Isso é **aceitável apenas para validação interna** — não apresentar para Techint como solução final sem resolver o ponto abaixo.

**Fase 2 — On-Premise (4-6 semanas adicionais, target: 2026-07-18):**
- **Decisão crítica — LLM local vs. Claude API com wrapper:**
  - **Opção A (recomendada para Fase 2):** Substituir Claude API por modelo local via Ollama (ex: Llama 3.1 70B) rodando no servidor do cliente. Dados nunca saem. Latência maior (~30s/doc), mas privacidade total.
  - **Opção B (curto prazo):** Manter Claude API mas com contrato de privacidade Anthropic (BAA/DPA) + processamento sem retenção de dados. Mais rápido de implementar, mas depende de aceitar a política Anthropic e comunicar isso para o cliente.
  - Decisão final depende do requisito de compliance da Techint — perguntar diretamente a Fúlvio antes de construir.
- Containerizar em Docker
- Script de instalação na rede interna do cliente
- Positioning: "Roda dentro da sua empresa — seus dados nunca saem" (válido apenas com Opção A)
- Pricing: R$30-50k para piloto Fase 1. R$100-300k/ano é target para Ano 2 após provar valor no piloto — não assumir esse valor na negociação inicial.

**Por que esse caminho?** Endereça diretamente o único bloqueio real nomeado por todos os prospects: "IA vai vazar nossos dados." Sem endereçar isso, nenhuma EPC assina. Com isso endereçado, o pricing pode ser premium e o lock-in é real.

## Open Questions

1. Techint tem poder para contratar sem concorrência pública ou precisa de processo licitatório?
2. O empregador do founder vai reivindicar propriedade intelectual se o JulIA for vendido externamente?
3. Quais outros tipos de equipamento (além de transmissores de pressão) têm maior volume nos projetos da Techint?
4. O template de PATEC é padronizado entre EPCs ou cada empresa tem o seu? (Isso afeta o esforço de onboarding por cliente)

## Success Criteria

- Fúlvio usa o protótipo em um projeto real e diz: "Eu mandaria esse parecer para o fornecedor"
- Techint assina um piloto pago (mínimo R$30-50k)
- Tempo de geração do PATEC no piloto < 30 minutos por instrumento (vs. horas manual)
- Zero incidentes de dados expostos

## Distribution Plan

- **Fase 1 (piloto):** Acesso direto via instância hospedada pelo founder, apenas para validação com Fúlvio. Sem instalação no cliente ainda.
- **Fase 2 (on-premise):** Container Docker + script de instalação no servidor do cliente. Suporte de setup incluído no contrato. Sem portal de management ainda — uma instalação por vez.
- **Pipeline de CI/CD:** On-premise via release de container versionado no GitHub Container Registry (ghcr.io). A versão Railway/Vercel existente serve apenas como sandbox de desenvolvimento — não chamar de "SaaS" para clientes neste estágio.
- **Distribuição inicial:** 100% via relacionamento direto. Alexandre contata os coordenadores que conhece. Sem marketing, sem site público, sem freemium.

## Dependencies

- Template PATEC da Techint (blocker para Fase 1 — sem isso, o output não tem valor)
- Dataset real anonimizado para validação (conseguir com Fúlvio)
- Decisão sobre propriedade intelectual com empregador (blocker para venda externa)
- Claude API key com limites de rate adequados para uso em produção
